Introduction

Welcome to the AI API documentation. This API provides access to state-of-the-art language models for chat completions, text generation, embeddings, and more.

The API is organized around REST. Our API has predictable resource-oriented URLs, accepts JSON-encoded request bodies, returns JSON-encoded responses, and uses standard HTTP response codes, authentication, and verbs.

Base URL
https://api.ai-service.com/v1

Base URL

All API requests should be made to the following base URL. HTTPS is required for all API communications.

EnvironmentBase URLDescription
Production https://api.ai-service.com/v1 Live environment for production workloads
Sandbox https://sandbox.api.ai-service.com/v1 Test environment with rate-limited free usage

Authentication

The AI API uses API keys to authenticate requests. You can view and manage your API keys in the Dashboard.

Your API keys carry many privileges, so be sure to keep them secure. Do not share your secret API keys in publicly accessible areas such as GitHub, client-side code, and so forth.

Security Warning
All API requests must be made over HTTPS. Calls made over plain HTTP will fail. API requests without authentication will also fail.

Authorization Header

Authenticate your API requests by including your API key in the Authorization header.

GET /auth/verify

Verify that your API key is valid and retrieve basic account information.

HeaderTypeDescription

Content-Type string Must be application/json

Chat Completions

Given a list of messages comprising a conversation, the model will return a response. This is the core endpoint for building chat-based applications.

POST /chat/completions

Creates a model response for the given chat conversation. Supports streaming for real-time token delivery.

ParameterTypeDescription
modelrequired string ID of the model to use. Example: gpt-4o, claude-3-opus
messagesrequired array A list of messages comprising the conversation. Each message has role (system/user/assistant) and content.
temperature number Sampling temperature between 0 and 2. Higher values make output more random. Default: 1.0
max_tokens integer Maximum number of tokens to generate. Default: model-dependent
top_p number Nucleus sampling parameter. Alternative to temperature. Default: 1.0
stream boolean If true, partial message deltas will be sent as SSE events. Default: false
tools array List of tools the model may call. Each tool requires type and function definition.
Response
200 OK

Returns a chat completion object or a stream of completion chunks if stream: true.

Streaming

When stream: true is set, the API returns server-sent events (SSE). Each event contains a partial completion chunk. The stream ends with a [DONE] message.

SSE Format
Each line starts with data: followed by a JSON object. Empty lines separate events.

Message Object

Messages are the primary input to the chat completions API.

FieldTypeDescription
rolerequired string One of system, user, assistant, or tool
contentrequired string The content of the message. Can be text or an array of content parts for multimodal inputs.
name string An optional name for the participant. Provides the model information to differentiate between participants.

Error Handling

The API uses conventional HTTP response codes to indicate the success or failure of an API request. Codes in the 2xx range indicate success, codes in the 4xx range indicate an error that failed given the information provided, and codes in the 5xx range indicate an error with the API servers.

Error Response Format

All errors follow a consistent JSON structure.

Error Object
{ "error": { "type": "invalid_request", "message": "...", "code": "invalid_api_key" } }

Error Codes

CodeHTTPDescription
invalid_api_key 401 The provided API key is invalid or revoked.
insufficient_quota 429 You have exceeded your rate limit or quota. Check your plan.
invalid_request 400 The request body is malformed or missing required fields.
model_not_found 404 The requested model does not exist or you do not have access.
server_error 500 An unexpected error occurred on our servers. Retry with backoff.

Retry Strategy

For 5xx errors and 429 rate limit responses, implement an exponential backoff retry strategy.

  • Retry immediately on 429 if Retry-After header is present
  • Use exponential backoff starting at 1 second, doubling each retry
  • Maximum retry delay: 60 seconds
  • Maximum retry attempts: 5
